# Phase 2d — Performance & Stability Overhaul Implementation Plan

## Objective
Address critical freezes reported by the user and fix match stability issues where players cannot join or get stuck on the "Waiting" screen.

## Problems Identified
1.  **Rendering Bottleneck**: `PixiCanvas` clones the entire cell map and `MapRenderer` clears/re-draws every cell every tick (30 FPS). With 1,600 cells, this is thousands of redundant draw operations per second.
2.  **Match Stability**: Player IDs are generated by the Gateway on every connection. A page refresh creates a "third" player from the Game Match's perspective, which is ignored because the match is "full" (2 players).
3.  **Gateway Robustness**: Unhandled WebSocket errors can crash the Gateway, disconnecting all players.

## Proposed Solutions

### 1. Stability: Persistent Player IDs
- **Web**: Store a unique `playerId` in `localStorage`.
- **Web**: Pass this `playerId` in the WebSocket connection URL (e.g., `ws://...?matchId=M&playerId=P`).
- **Gateway**: Extract `playerId` from the query string. If missing, fallback to a new UUID.
- **Game Match**: In `addPlayer`, if the ID is already in the match, just re-send the `init` message and continue. This makes refreshes seamless.

### 2. Performance: Optimized Map Rendering
- **Web**: Refactor `MapRenderer` to use a single `Graphics` object for the static map.
- **Web**: Refactor `updateCells` to only draw the specific cells that changed, rather than re-drawing the whole map.
- **Web**: Refactor `PixiCanvas` store subscription to track the `tick` and only pass `diff.cellsChanged` to the renderer.

### 3. Performance: Server-side Hoisting
- **Game Server**: Safely restore the optimization to pre-calculate `buriedPositions` and `playersSnapshot` once per tick, rather than once per player per tick.

## Implementation Steps

  - Extract `playerId` from query params.
  - Remove UUID generation from the connection handler.

### Task 2: Game Match Stability
- [ ] **Step 1: `GameMatch.ts`**
  - Update `addPlayer` to handle re-joining existing IDs even if `players.size === 2`.
  - Ensure `emitInit` is always called for the re-joining player.

### Task 3: Map Rendering Optimization
- [ ] **Step 1: `MapRenderer.ts`**
  - Refactor to use a single `mapGfx` object for all cells.
  - `initMap` draws all cells once.
  - `updateCells` takes an array of changes and surgically draws them.
- [ ] **Step 2: `PixiCanvas.tsx`**
  - Track `lastTick` in a `useRef`.
  - Inside the subscription, only call `updateCells` if the tick has changed and there are actual changes in the store's "last diff" (might need to expose the last diff in the store or just pass the whole state and let renderer decide).
  - Actually, `gameStore.applyDiff` already has `cellsChanged`. We can expose the `lastCellsChanged` in the store.

### Task 4: Server Performance Refactor (Restore)
- [ ] **Step 1: `GameMatch.ts`**
  - Move `buriedPositions` and `playersSnapshot` out of the loop.
